Installing WSL

First off we want to pop into PowerShell to get WSL2 installed on the Windows machine. Right-click the start menu icon and click ‚ÄúWindows PowerShell (Administrator)‚Äù or use the Win-X + A shortcut to do the same thing.

Next, run the command wsl --install to quickly install everything needed to run Ubuntu on Windows. After that has finished installing, in PowerShell, type the Restart-Computer cmdlet to restart and finish installing WSL2.

A terminal should open on restart with the text Installing, this may take a few minutes... After this completes, you will need to type in a username and password for your WSL virtual machine. You will need this password to run sudo and other privileged commands in WSL.

Installing Necessary Development Packages

After we get a terminal open, run the following commands to update Ubuntu package information and upgrade all installed software:

 sudo apt update && sudo apt upgrade -y

After this completes, run the following command to install git, the gcc toolchain, gdb, and cmake. 

sudo apt install git git-extras build-essential gcc-multilib gdb cmake -y

Git and SSH Key Setup

Next, it will be helpful if we didn‚Äôt have to type in a token or a password each time we want to pull or push to our Git remotes. Therefore, we gotta have SSH keys to git gud [sic].

In the terminal, generate a ssh key by using the following command (replace your email first): 

ssh-keygen -t rsa -b 4096 -C "email@gmail.com"

You will need to press enter to save in the default location, and don‚Äôt put in a password unless you want to type it in all the time (an extra layer of security if you want it).

The output should look like this (if you want the fancy tabbed terminal, install Terminal from MS Store)

Next, run the following commands to register the ssh key: 

eval "$(ssh-agent -s)"; ssh-add ~/.ssh/id_rsa

Home stretch, now need to add the SSH key to GitHub. Run the following command to print your public key to stdout: cat ~/.ssh/id_rsa.pub

Copy the entire output (including ssh-rsa) to your clipboard.

Pop over to GitHub and navigate to Settings->SSH & GPG Keys. Click ‚ÄúNew SSH Key‚Äù. Name it something descriptive, like ‚ÄúDell Laptop‚Äù or something that ties it to your machine. Paste the key into the Key field and click ‚ÄúAdd SSH Key‚Äù.

Go to the terminal now and run the following commands to configure git: 

git config --global user.email "yourgithubemail@email.com" 
git config --global user.name "Firstname Lastname"

Now, you should be able to clone, push & pull from Github without a password.

Install VS Code and Extensions

Install VS Code for Windows (don‚Äôt install it in the terminal). After it is installed and open, close & reopen the terminal to make sure the PATH environment variable is updated to include VS Code.

In VS Code, click Extensions on the left hand navigation bar, then type in C/C++ Extension Pack to install the necessary extensions. (It should have 9 extensions included, not just the C/C++ extension).

Next, hop over into the terminal and navigate to a repo you have cloned. (If you need to clone a repo for a lab or something, go to the repository in Github and click ‚ÄúCode‚Äù in the top right, then SSH, then copy that link.

Pop back over to your trusty terminal. You probably don‚Äôt want to clone repos into your home folder in the terminal, so run the following commands to make a dev folder in your home directory.

cd ~ # CD to home directory (/home/username is equivalent to ~)
mkdir dev # You should probably know what these commands do by now :)
cd dev

and run the following command with the link you just copied.

git clone PASTE-LINK-HERE

Now cd into the repo you cloned and run the following command to open VS Code.

code .

If it is working properly, you should see similar output:

Almost done with the WSL setup, now we just need to install the WSL Extensions inside of the WSL VM. After VS Code comes up, you should see the text ‚ÄúWSL: Ubuntu‚Äù in the bottom left corner.

Go back to the extensions tab, scroll down until you see C/C++ Extension Pack and then click ‚ÄúInstall in WSL: Ubuntu‚Äù.

We won‚Äôt use CMake, so disable CMake and CMake tools in the local extensions.

Build Tasks and Debugging

Setup Build Tasks

Build tasks are roughly equivalent to the make files that we use in class (e.g. ./mk).

However, they are written in .JSON format instead of bash scripts. To get started, make sure you have your project open in VS Code by using cd to navigate to your repo folder and calling the code . command. (Do this from WSL).

Now make you have your code open in the editor, press F5 or click Run‚ÜíStart Debugging so we can generate our tasks.json and launch.json files. It will ask you to select an environment: choose C++: GDB/LLDB and then choose gcc - Build and debug active file. This will start debugging, but go ahead and stop the debugger by clicking the red stop button in the top bar.

In the file explorer on the left, click tasks.json and you should see something like this:

  1. First, we want to change ${file} because this would only build the current file (t.c). Sometimes, we need to link multiple .c files (like the command gcc t1.c t2.c t3.c links the 3 .c files). To automatically link any .c files in the folder, replace "${file}" with "${fileDirname}/*.c"
  1. We want to replace "${fileDirname}/${fileBasenameNoExtension}" with "${fileDirname}/a.out" because the Auto Grader GitHub script will look for a file with this name.

Don‚Äôt forget to save the file! (Ctrl+S).

It should look like this now:

This task is now equivalent to running the following command:

gcc -fdiagnostics-color=always -g ./*.c -o ./a.out

If you need to add another option for gcc, you can do so by adding to this JSON file in the args array.

Now open launch.json from the explorer. Edit "${fileDirname}/${fileBasenameNoExtension}‚Äù so that the value is "${fileDirname}/a.out‚Äù. This makes it so that the debugger (gdb) can find the a.out file that we create with the build task. It should now look like this:

Last thing, make sure to delete the executable file in your build directory. (In my case, it is just called t because it took the name from t.c. This isn‚Äôt necessary because we will be using the name a.out.

Debugging

FINALLY, we can start debugging our programs!

  1. Create a breakpoint in your C code by clicking to the left of the line numbers (1).
  1. Start debugging by pressing F5 or clicking the little bug shown in (2)
  1. It will ask you to choose a build task, choose gcc - Build and debug active file

Features of Debugger

  1. This is the toolbar where you can continue, step over, step in, step out, or stop debugging
  1. This is the variables and registers available when debugging. If you want to look at globals, add them to the Watch section (3)
  1. This is the Watch section. If you want to look at globals, add them here by clicking the plus sign and typing in an expression (variable name, function call, basically any expression.)

There is also cool stuff like conditional breakpoints, the Call Stack visualizer, and other advanced features you can look into.

Pushing Changes Back to GitHub

When your code is ready to go, you can use the integrated terminal in VS code to push your changes to GitHub. First, add all the changes in your repo to the staging area by running this command:

git add .

Next, we want to make a commit (or saved snapshot of these changes). You must enter a comment with these commits. Follow the example below:

git commit -m "Finished lab, definitely not buggy code I promise!!!1"

Finally, push it to the remote (GitHub‚Äôs servers in this case) by running the following command:

git push

That‚Äôs it, your changes are now stored in GitHub, and your TA‚Äôs can grade your code.
